# CNV 4.99.0-2739 nightly: QEMU machine type mismatch blocks all VMIs

**Date:** 2026-04-21
**Updated:** 2026-04-22
**CNV version:** 4.99.0-2739 (nightly)
**OCP version:** 4.22.0-rc.0
**RHCOS:** 9.8.20260417-0 (kernel 5.14.0-687.5.1.el9_8)
**Platform:** ARO (Azure Red Hat OpenShift), centralus
**Node VM size:** Standard_D8s_v5 (Dsv5, nested virt supported)

## Summary

The default `machineType` in KubeVirt (`pc-q35-rhel9.8.0`) is not supported by the QEMU shipped in the virt-launcher container of CNV 4.99.0-2739. This prevents all VMIs from running.

**Workaround:** Override the machine type to `pc-q35-rhel9.6.0` via HCO jsonpatch annotation (see below). After applying this, the validation checkup runs successfully.

A separate cosmetic issue (missing `clientcertificates` volume mount) causes recurring log errors in virt-handler but does not block VMI creation.

## Issue 1: QEMU does not support pc-q35-rhel9.8.0

### Symptoms

virt-handler fails to sync any VMI:

```json
{
  "msg": "Synchronizing the VirtualMachineInstance failed.",
  "reason": "server error. command SyncVMI failed: \"LibvirtError(Code=67, Domain=10,
    Message='unsupported configuration: Emulator '/usr/libexec/qemu-kvm' does not
    support machine type 'pc-q35-rhel9.8.0'')\""
}
```

### Analysis

- KubeVirt `spec.configuration.architectureConfiguration.amd64.machineType` is set to `pc-q35-rhel9.8.0`.
- The RHCOS host is 9.8, so the host QEMU should support this machine type.
- However, the **virt-launcher container's QEMU** does not include `pc-q35-rhel9.8.0` in its supported machine types.
- Node labels confirm this: machine-type labels go up to `pc-q35-rhel9.6.0` only. The `pc-q35-rhel9.8.0` label is absent on all worker nodes.
- This means the virt-launcher container image in this nightly build ships a QEMU version that predates RHEL 9.8 machine type support, yet the KubeVirt config already references 9.8.

### Worker node machine-type labels (all three workers identical)

```
machine-type.node.kubevirt.io/pc-q35-rhel9.6.0=true   (highest)
machine-type.node.kubevirt.io/pc-q35-rhel9.4.0=true
machine-type.node.kubevirt.io/pc-q35-rhel9.2.0=true
machine-type.node.kubevirt.io/pc-q35-rhel9.0.0=true
machine-type.node.kubevirt.io/pc-q35-rhel8.6.0=true
... (older types present)
```

**Missing:** `machine-type.node.kubevirt.io/pc-q35-rhel9.8.0=true`

### Impact

- All VMIs fail at the libvirt level.
- The validation checkup job cannot run any tests.

### Workaround

Override the default machine type via HCO jsonpatch annotation:

```bash
oc annotate --overwrite hyperconverged kubevirt-hyperconverged -n openshift-cnv \
  'kubevirt.kubevirt.io/jsonpatch=[{"op":"add","path":"/spec/configuration","value":{"architectureConfiguration":{"amd64":{"machineType":"pc-q35-rhel9.6.0"}}}}]'
```

This sets the amd64 machine type to `pc-q35-rhel9.6.0` (the highest type supported by the virt-launcher's QEMU). HCO propagates the change to the KubeVirt CR. Patching the KubeVirt CR directly does not persist — HCO reverts it.

Per maintainer feedback, the certificate error (Issue 2 below) is a red herring and the machine type mismatch is the actual blocker.

## Issue 2: Missing clientcertificates volume mount (cosmetic)

### Symptoms

virt-handler logs show a recurring error every 60 seconds:

```
failed to load the certificate /etc/virt-handler/clientcertificates/tls.crt
  and /etc/virt-handler/clientcertificates/tls.key
reason: open /etc/virt-handler/clientcertificates/tls.crt: no such file or directory
```

### Analysis

The virt-handler daemonset mounts three certificate volumes:

| Volume | Mount Path |
|--------|-----------|
| `kubevirt-virt-handler-server-certs` | `/etc/virt-handler/servercertificates` |
| `kubevirt-virt-handler-migration-client-certs` | `/etc/virt-handler/migrationservercertificates` |
| `kubevirt-virt-handler-vsock-client-certs` | `/etc/virt-handler/vsockclientcertificates` |

The code (cert-manager.go:189) expects a fourth path `/etc/virt-handler/clientcertificates/` which has **no corresponding volume or volume mount** in the daemonset spec.

### Impact

- Cosmetic log noise (error every 60s per virt-handler pod)
- virt-handler has 3 restarts per pod but continues to function
- Does **not** block VMI creation or scheduling (per maintainer confirmation)

## Validation checkup results (after workaround)

After applying the machine type override, the validation checkup completed successfully:

| Suite | Ran | Passed | Failed | Pending | Skipped |
|-------|-----|--------|--------|---------|---------|
| **Compute** | 63 | **63** | **0** | 0 | 1319 |
| **Network** | 11 | **9** | **2** | 4 | 1367 |
| **Storage** | 71 | **38** | **33** | 4 | 1307 |

### Network failures (2)

1. **test_id:1514** — `Networkpolicy: should fail pinging between two VMI/s each on different namespaces`
   - Cross-namespace ping was expected to fail (NetworkPolicy enforcement) but succeeded.
   - Possible OVN-Kubernetes / ARO network policy enforcement issue.
   - Error: `Timed out after 18.174s. Expected failure, but got no error.`

2. **test_id:1780** — `Networking: should be able to reach the outside world [IPv4]`
   - VM failed to ping `8.8.8.8`.
   - **Root cause: Azure blocks outbound ICMP.** Verified by running a privileged pod on the cluster — `ping 8.8.8.8` gets 100% packet loss, while `curl https://8.8.8.8` succeeds (HTTP 302). This is an Azure platform limitation, not an ARO or CNV issue. The cluster is not private and has working outbound TCP/UDP connectivity.
   - Error: `failed to ping VMI ... error: ping -c 5 -w 15 8.8.8.8 failed`

### Storage failures (33)

- **25 failures** are `RequiresRWXBlock` tests — expected, as the cluster only has `managed-csi` (Azure Disk, RWO). No RWX storage class is available.
- **8 failures** are `Export` tests — PVC never becomes Bound during "Creating source volume" step. The `managed-csi` StorageClass uses `WaitForFirstConsumer` binding mode; the export test creates a standalone PVC without a consumer pod, so it stays Pending and times out.

## Reproduction

1. Deploy ARO 4.22.0-rc.0
2. Install CNV nightly 4.99.0-2739 from `registry-proxy.engineering.redhat.com`
3. Create any VMI (or run the ocp-virt-validation-checkup)
4. Observe: VMI fails with LibvirtError for unsupported machine type
5. Apply the machineType workaround above
6. Re-run — VMIs now start successfully
