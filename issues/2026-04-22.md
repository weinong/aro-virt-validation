# MSHV (hyperv-direct) validation blocked: RHCOS kernel lacks L1VH partition support

**Date:** 2026-04-22
**Updated:** 2026-04-23
**CNV version:** 4.99.0-2739 (nightly)
**OCP version:** 4.22.0-rc.0
**RHCOS:** 9.8.20260417-0 (kernel 5.14.0-687.5.1.el9_8)
**Platform:** ARO (Azure Red Hat OpenShift), centralus
**Node VM size:** Standard_D192ds_v6

## Summary

Attempted to validate the MSHV (Microsoft Hypervisor) / `hyperv-direct` feature in KubeVirt on ARO. The goal was to run KubeVirt VMs using the MSHV hypervisor instead of KVM, which requires the node to boot in an L1VH (Level-1 Virtual Hypervisor) partition.

**Result:** The node does not boot in an L1VH partition with RHCOS 9.8. The RHCOS kernel (5.14.0-687.5.1.el9_8) does not have MSHV guest / L1VH awareness. L1VH support requires a newer kernel with MSHV patches (typically 6.x+). **Resolved** by upgrading to RHCOS 10.2 (kernel 6.12.0) via the OS Streams Tech Preview — see [Resolution](#resolution-upgrade-to-rhcos-102-via-os-streams-tech-preview) below.

## Steps Taken

### 1. Machineset with D192ds_v6 SKU

Created a new machineset with `Standard_D192ds_v6` (192 vCPU) and the Azure VM tag:

```
platformsettings.host_environment.nodefeatures.hierarchicalvirtualizationv1 = True
```

This tag requests Azure to place the VM on a host with nested/hierarchical virtualization support.

**Quota issues:** The initial Ddsv6 family quota (10 vCPUs) and total regional quota (116 vCPUs) were insufficient. Raised to 400 and 500 respectively.

### 3. dmesg Analysis

The node boots as a standard Hyper-V guest, not in L1VH mode:

```
Hypervisor detected: Microsoft Hyper-V
Hyper-V: Nested features: 0x0
Booting paravirtualized kernel on Hyper-V
```

- `Nested features: 0x0` — no nested virtualization features exposed
- No "l1vh partition" string found in dmesg
- Kernel is `5.14.0-687.5.1.el9_8` (RHEL 9.8, based on upstream 5.14)

## Root Cause

The RHEL 9.8 kernel (5.14.x) shipped with RHCOS does not include MSHV guest patches required for L1VH partition awareness. Even when placed on an L1VH-capable Azure host (via the platform tag), the kernel does not recognize or enter the L1VH partition mode. L1VH support requires a kernel with MSHV patches, typically based on upstream 6.x or later.

## Resolution: Upgrade to RHCOS 10.2 via OS Streams (Tech Preview)

The OCP 4.22 payload ships both RHCOS 9.8 (`rhel-coreos`) and RHCOS 10.2 (`rhel-coreos-10`). RHCOS 10.2 includes kernel 6.12.0 which has MSHV/L1VH partition support. Using the OS Streams Tech Preview feature, we upgraded only the MSHV nodes to RHCOS 10.2 while keeping all other nodes on RHCOS 9.8.

Reference: [How to test RHCOS 10 on OpenShift 4.21 and 4.22](https://access.redhat.com/articles/7139678)

### Step 1: Enable TechPreviewNoUpgrade

> **WARNING:** This is irreversible. The cluster will no longer be eligible for future minor version upgrades.

```bash
oc patch featuregate/cluster --type merge -p '{"spec":{"featureSet":"TechPreviewNoUpgrade"}}'
```

Wait for cluster operators to stabilize (~1-2 minutes in our case). The MCO will restart and register the `OSImageStream` CRD.

### Step 2: Confirm rhel-10 stream is available

```bash
oc get osimagestreams/cluster -o yaml
```

Expected output should show both `rhel-9` and `rhel-10` under `status.availableStreams`:

```yaml
status:
  availableStreams:
  - name: rhel-10
    osImage: quay.io/openshift-release-dev/ocp-v4.0-art-dev@sha256:4738bcbc...
  - name: rhel-9
    osImage: quay.io/openshift-release-dev/ocp-v4.0-art-dev@sha256:36686f5c...
  defaultStream: rhel-9
```

### Step 3: Create a dedicated MachineConfigPool for MSHV nodes

To avoid upgrading all workers to RHCOS 10, create a separate MCP that selects nodes with the `node-role.kubernetes.io/mshv` label:

```yaml
apiVersion: machineconfiguration.openshift.io/v1
kind: MachineConfigPool
metadata:
  name: mshv
spec:
  machineConfigSelector:
    matchExpressions:
      - key: machineconfiguration.openshift.io/role
        operator: In
        values: [worker, mshv]
  nodeSelector:
    matchLabels:
      node-role.kubernetes.io/mshv: ""
  osImageStream:
    name: rhel-10
```

The `osImageStream.name: rhel-10` tells the MCO to use RHCOS 10 for nodes in this pool.

### Step 4: Create the MSHV machineset with the mshv role label

The machineset must set both `node-role.kubernetes.io/worker` and `node-role.kubernetes.io/mshv` labels on the node, and include the Azure VM tag for L1VH host placement:

```bash
oc get machineset <existing-worker-ms> -n openshift-machine-api -o json \
  | jq '
    del(.metadata.uid, .metadata.resourceVersion, .metadata.creationTimestamp,
        .metadata.generation, .status, .metadata.annotations) |
    .metadata.name = "<cluster-id>-worker-mshv-centralus1" |
    .spec.replicas = 1 |
    .spec.selector.matchLabels."machine.openshift.io/cluster-api-machineset" = "<cluster-id>-worker-mshv-centralus1" |
    .spec.template.metadata.labels."machine.openshift.io/cluster-api-machineset" = "<cluster-id>-worker-mshv-centralus1" |
    .spec.template.spec.providerSpec.value.vmSize = "Standard_D192ds_v6" |
    .spec.template.spec.providerSpec.value.osDisk.diskSizeGB = 256 |
    .spec.template.spec.providerSpec.value.zone = "1" |
    .spec.template.spec.providerSpec.value.tags = {
      "platformsettings.host_environment.nodefeatures.hierarchicalvirtualizationv1": "True"
    } |
    .spec.template.spec.metadata.labels."node-role.kubernetes.io/mshv" = "" |
    .spec.template.spec.metadata.labels."node-role.kubernetes.io/worker" = ""
  ' | oc apply -f -
```

Key points:
- **`tags`** must be a `map[string]string` (not an array) in the Azure providerSpec
- The `node-role.kubernetes.io/mshv` label causes the node to join the `mshv` MCP
- The Azure tag `hierarchicalvirtualizationv1=True` requests L1VH-capable host placement

### Step 5: Wait for the node to be ready and MCP to finish rolling out

The node will initially boot with RHCOS 9.8 (from the marketplace image), then the MCO will reboot it into RHCOS 10.2 via the mshv MCP. Monitor progress:

```bash
# Watch machine provisioning
oc get machines -n openshift-machine-api -o wide

# Watch MCP rollout
oc get mcp mshv -w
```

The MCP will show `UPDATING=True` while the node is being rebased to RHCOS 10.2, then `UPDATED=True` when complete.

### Step 6: Verify L1VH partition mode

```bash
oc debug node/<mshv-node-name> -- chroot /host dmesg | grep -i "l1vh"
```

Expected output:

```
Hyper-V: running as L1VH partition
```

### Result

After upgrading to RHCOS 10.2:

| Property | RHCOS 9.8 (before) | RHCOS 10.2 (after) |
|---|---|---|
| Kernel | 5.14.0-687.5.1.el9_8 | 6.12.0-211.7.1.el10_2 |
| OS Image | RHCOS 9.8.20260417-0 (Plow) | RHCOS 10.2.20260408-0 (Coughlan) |
| L1VH partition | Not detected | `Hyper-V: running as L1VH partition` |
| Container runtime | cri-o://1.35.2-9 (el9) | cri-o://1.35.2-2 (el10) |
