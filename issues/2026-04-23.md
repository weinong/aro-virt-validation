# HCO patch and validation: hyperv-direct on MSHV/L1VH

**Date:** 2026-04-23
**CNV version:** 4.99.0-2759 (nightly, auto-upgraded from 4.99.0-2739)
**OCP version:** 4.22.0-rc.0
**Cluster:** aro-virt-test

## Summary

VMs created under the `hyperv-direct` hypervisor crash immediately at boot.
The root cause is that the RHCOS 10.2 kernel's `mshv_root` module rejects
`MSHV_CREATE_PARTITION` ioctls as "unsupported", despite the node running in
L1VH partition mode.

## Blocking issue: MSHV_CREATE_PARTITION ioctl unsupported

### Symptoms

### L1VH status

```
Hyper-V: running as L1VH partition
Hyper-V: Nested features: 0x0
```

`Nested features: 0x0` is **expected** for the MSHV/L1VH path. This value
represents `CPUID[4000000Ah].EAX` and does not impact MSHV operation. When
using MSHV with a nested hypervisor, the guest OS is a dom0 L2 root partition
running on a nested hypervisor â€” traditional nested virt CPUID flags are not
needed.

The blocking issue is the `MSHV_CREATE_PARTITION` ioctl being rejected by the
kernel module.

## HCO patch

The HCO jsonpatch for `hyperv-direct` was successfully applied after fixing
the field name from `hypervisorConfiguration` (object) to `hypervisors` (array).

### cpuModel issue (fixed)

Setting `cpuModel: qemu64-v1` in the HCO config causes KubeVirt to add
`cpu-model.node.kubevirt.io/qemu64-v1=true` as a nodeSelector on every VMI
launcher pod. Since `qemu64-v1` is a virtual QEMU CPU model (not a real CPU
model that virt-handler advertises on nodes), VMIs could never schedule.

**Fix:** Removed `cpuModel` and `obsoleteCPUModels` from the HCO jsonpatch.

## mshv_root module loading (fixed)

The `mshv_root` kernel module is not auto-loaded at boot. A MachineConfig
(`99-mshv-root-module`) was created to load it via `/etc/modules-load.d/mshv-root.conf`.

The MCO drain was stuck because this is a single-worker-node cluster â€” PDBs on
single-replica deployments (image-registry, router, prometheus, alertmanager, etc.)
prevent eviction when there's no other node to reschedule to. Fixed by force-deleting
the blocking pods to allow the drain and reboot to proceed.

## Previous validation results are BOGUS

The validation checkup tool (ocp-virt-validation-checkup) reports incorrect results.
When the Ginkgo `SynchronizedBeforeSuite` fails, the tool still reports the
_intended_ test count as "passed":

| What the tool reports | What actually happened |
|----------------------|----------------------|
| 1019 passed, 0 failed | `Ran 0 of 1380 Specs` â€” zero tests ran |
| 152 passed, 0 failed | `Ran 0 of 1380 Specs` â€” zero tests ran |

The `SynchronizedBeforeSuite` failure was: "Both mshv and vhost-net devices are
required for testing, but are not present on cluster nodes" â€” because `/dev/mshv`
did not exist until `mshv_root` was loaded via MachineConfig.

After fixing the module loading, the BeforeSuite passes but VMs crash (see above).
