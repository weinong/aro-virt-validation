# CNV 4.99.0-2759: hypervisorConfiguration not in KubeVirt CRD

**Date:** 2026-04-23
**CNV version:** 4.99.0-2759 (nightly, auto-upgraded from 4.99.0-2739)
**OCP version:** 4.22.0-rc.0
**Cluster:** aro-virt-test

## Summary

The `hypervisorConfiguration` field is **not present** in the KubeVirt CRD shipped with CNV 4.99.0-2759. The HCO jsonpatch annotation for `hyperv-direct` is accepted but the field is silently dropped during reconciliation — it never appears in the KubeVirt CR.

## Evidence

The `ConfigurableHypervisor` feature gate is accepted:

```
$ oc get kubevirt kubevirt-kubevirt-hyperconverged -n openshift-cnv \
    -o jsonpath='{.spec.configuration.developerConfiguration.featureGates}'
["ConfigurableHypervisor","CPUManager","Snapshot","HotplugVolumes",...]
```

But the CRD does not define `hypervisorConfiguration`:

```
$ oc get crd kubevirts.kubevirt.io -o json | python3 -c "
import json,sys
d=json.load(sys.stdin)
props=d['spec']['versions'][0]['schema']['openAPIV3Schema']['properties']['spec']['properties']['configuration']['properties']
print('hypervisorConfiguration' in props)
"
False
```

The HCO annotation is set correctly but has no effect:

```
$ oc get kubevirt kubevirt-kubevirt-hyperconverged -n openshift-cnv -o yaml | grep hypervisor
(no output)
```

## Other fields applied successfully

## Impact

Without `hypervisorConfiguration.name=hyperv-direct`, VMs will launch using the default KVM/QEMU hypervisor even on the L1VH node. The `hyperv-direct` code path in KubeVirt is not yet wired up in this nightly build.
